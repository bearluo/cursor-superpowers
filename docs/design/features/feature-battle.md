# 战斗模块设计（MVP）

## 1. 背景与目标

战斗模块是本项目“爽点密度”和“风险押注”同时成立的主舞台：一方面要让玩家持续获得构筑变强的反馈，另一方面要将腐化与过载的代价具象化，形成可预期、可承担的风险决策。

MVP 阶段聚焦三项目标：
- 构筑可感知：玩家能理解协议叠层与反应触发，并有意识地走向清屏/控场/生存流派。
- 节奏可推进：10-15 分钟单局内形成“前期成型 -> 中期押注 -> 后期冲刺”的曲线。
- 结果可验证：反应、腐化阈值、过载衰减、失控事件均具备可观测反馈与可复现验收路径。

## 2. 核心对象与规则定义

- 协议（Protocol）：战斗元素载体。MVP 包含热、冷、毒、电四类。
- Stack：目标上的协议层数（0-100），用于决定反应触发条件。
- 反应（Reaction）：满足条件时由多协议组合触发的玩法事件，不等同于纯伤害加成。
- 腐化（Corruption）：局内风险值，跨阈值会触发失控事件。
- 过载（Overload）：主动开启的短时强化窗口，收益与代价同时放大。
- 失控事件（Meltdown Event）：高腐化时注入战场的突发机制，需同时包含压力与回报。
- 守门者（Gatekeeper）：阶段性首领，用于检验构筑与风险管理能力。

## 3. 战斗时序与状态流

1. 输入与命中阶段：攻击/技能命中后写入或刷新协议 Stack。  
2. 反应判定阶段：检查协议组合、阈值与过载修正，满足即触发反应。  
3. 结算与反馈阶段：处理伤害、控制、扩散、连锁，并输出统一反馈。  
4. 风险更新阶段：结算腐化增减；跨阈值时挂起失控事件请求。  
5. 事件注入阶段：按规则注入追猎者/反噬/环境异常，并提供前摇提示。  
6. 过载生命周期阶段：开启强化 -> 持续放大收益与风险 -> 结束施加衰减。  

### 3.1 敌人追击与刷怪职责边界（MVP 约束）

- 敌人 AI 保持最小化：**怪物继续沿玩家方向直线追击**，不引入包围槽位、预测拦截等复杂行为。
- 难度与压迫感主要由刷怪系统提供：通过**刷怪方向分布**与**时间增援节奏**塑造空间压力与时间压力。
- 设计原则：优先保持“简单爽刷”与可维护性，避免在 MVP 阶段引入高复杂度追击算法。

## 4. 机制细则（建议参数区间）

以下参数用于联调起点，后续按实测调整。

### 4.1 协议与 Stack
- Stack 范围：0-100，建议自然衰减 2-5/秒。
- 单次叠层：普通攻击建议 6-12，技能建议 15-30。
- 同源重复触发应设计递减收益，避免单一路径碾压。

### 4.2 反应触发
- 触发前提建议：目标同时具备两种协议且各自 Stack >= 20。
- 反应定义：
  - 热 + 电：过热放电（小范围爆发，连锁跳数 +1）
  - 冷 + 电：超导（范围减速 20%-35%，易伤 10%-20%）
  - 冷 + 热：热震（清空双 Stack，造成高额瞬时爆发）
  - 毒 + 热：燃毒（DOT 转扩散，建议半径 2.5-4.0m）
  - 毒 + 电：神经脉冲（连锁附带短麻痹 0.3-0.8s）
- 反应全局最短间隔建议 0.1-0.2s，避免同帧爆量触发。

### 4.3 腐化增长与抑制
- 时间基础增长建议：0.4-0.8/秒。
- 行为增量建议：
  - 受击：+1 至 +3
  - 高阶反应：+0.5 至 +1.5
  - 过载启动瞬间：+8 至 +15
  - 过载期间附加：+0.8 至 +1.6/秒
- 抑制来源建议：
  - 净化掉落：单次 -6 至 -15
  - 防御向模块：小幅持续抑制
- 阈值建议：33 / 66 / 100。

### 4.4 过载收益与衰减
- 持续建议：6-10 秒。
- 收益建议：降低反应阈值（如 20 降至 15）和/或提升反应倍率（20%-40%）。
- 衰减建议：2-4 秒，施加移速下降/攻速下降/承伤上升中的一种或组合。

### 4.5 失控事件注入
- 触发时机：跨阈值即时，或过载结束后腐化仍处高位。
- 事件持续建议：8-20 秒。
- 设计约束：必须“有压力也有收益”，避免纯惩罚体验。

### 4.6 刷怪压力模型（替代复杂 AI）

- 目标：在不提升怪物 AI 复杂度的前提下，解决“可持续 kite 慢磨”问题。
- 刷怪方向策略（推荐）：
  - 以玩家当前位置为中心，优先在**玩家移动前方扇区**（建议 ±35°~45°）生成怪物；
  - 保留少量侧后方生成（建议 15%-25%）避免模式单一。
- 刷怪距离策略（建议）：
  - `minSpawnRadius` 建议 180-220；
  - `maxSpawnRadius` 建议 280-340；
  - 禁止贴脸出生，避免无提示伤害。
- 时间增援（中等递增，推荐）：
  - clearing 阶段每 10-15 秒触发一次增援（建议 12 秒）；
  - 前 60 秒批次建议 `1 -> 2 -> 3` 平滑递增；
  - 60 秒后维持中高密度，避免强度断崖。
- 场上容量上限：
  - 建议设置 `aliveCap`（普通 120-140；守门者波可放宽到 140-160）；
  - 增援按剩余容量截断，避免失控堆积导致不可玩。

### 4.7 经验掉落与奖励触发（爽刷模式）

- 触发原则：
  - 奖励选择不再由“清怪即触发”直接驱动；
  - 改为“**击杀掉经验球 -> 玩家拾取经验 -> 达到阈值触发奖励**”。
- 奖励阈值（随等级增长）：
  - 奖励等级记为 `rewardLevel`，从 1 开始；
  - 阈值建议公式：`xpNeed(level) = round(8 + 3*level + 1.5*level^1.35)`；
  - 拾取经验支持溢出：`xpCurrent >= xpNeed` 时扣除阈值并保留剩余经验。
- 经验球档位（建议）：
  - `S`：`xp=1`
  - `M`：`xp=3`
  - `L`：`xp=8`
- 按怪物强度动态掉率（核心）：
  - 先定义强度系数 `power`（示例：普通 1.0、精英 1.8、守门者 3.0，可叠加波次修正）；
  - 基础权重：`S=60, M=30, L=10`；
  - 强度修正：`S' = S / power`, `M' = M * sqrt(power)`, `L' = L * power`；
  - 最终掉率由 `S' / M' / L'` 归一化得到。
- 设计约束：
  - 不做“保底补正球/防卡关兜底”；
  - 爽刷模式下允许经验高溢出，由玩家拾取路线与战斗决策共同决定奖励节奏。

## 5. 反馈与可读性要求

### 5.1 即时反馈（毫秒级）
- 命中反馈需标识协议归属（颜色或图标）。
- 反应触发至少展示两类信息：反应名/图标、范围提示、数值/状态浮字。
- 控制效果需有明显状态标记和剩余时长感知。

### 5.2 HUD 决策反馈（秒级）
- 常驻展示：协议倾向、当前腐化、下个阈值距离、过载状态。
- 阈值前需有预警提示，避免无提示惩罚。
- 过载开始与结束需有强提示，明确押注窗口与衰减窗口。

### 5.3 复盘调优反馈（调试级）
- 记录关键数据：反应触发次数、过载使用次数、阈值触发时间点。
- 最低复盘项：最大腐化值、失控事件类型与次数、主要死亡原因。

## 6. 联调与验收

### 6.1 最小可复现场景
1. 反应可触发性：60 秒内稳定触发至少 3 种不同反应。  
2. 风险节奏可感知：单局内至少跨越 2 个腐化阈值，并可识别对应事件。  
3. 过载押注闭环：完成“开启 -> 增益体验 -> 衰减代价”的完整循环。  
4. 守门者检验有效：通关与失败可归因，不依赖纯数值平推。  
5. 刷怪压迫有效：在“怪物直线追击”前提下，前方刷怪与时间增援可显著降低无限 kite 可行性。  

### 6.2 通过标准
- 反应反馈满足“至少两类提示”的可读性标准。
- 腐化阈值与失控事件触发逻辑一致，无无提示突发惩罚。
- 过载收益与代价同时可感知，不出现单边体验失衡。
- 敌人行为保持简单可维护（直线追击），主要难度来源于刷怪分布与增援节奏，而非复杂 AI。

## 7. 开放问题

- 过载资源最终采用冷却制、充能制，还是混合制？
- 首版反应调性优先偏爆发清屏，还是控场生存？
- 失控事件是否需要保护间隔，避免连续插入破坏操作流畅度？
