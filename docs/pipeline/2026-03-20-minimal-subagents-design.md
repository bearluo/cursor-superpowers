## 最小 Subagent 组合设计（A+C+D）

**日期**：2026-03-20  
**目标**：为团队在 Cursor/Agent 工作流中定义“最小子代理”集合与使用边界，在保证交付质量的同时尽量降低成本与返工。

### 核心取向
1. **A：省调用**  
   默认不并行派发多个子代理；同一时刻只维持一个主会话负责定位与改动。
2. **C：少返工**  
   在任何“宣称完成/可合并”之前，必须通过验证门禁（verification gate）。
3. **D：文档可执行**  
   规则需要写进文档，让团队成员能照单执行，不依赖个人习惯。

## 1. 最小 Subagent 集合（推荐）

### 必选概念：主编排代理（Main）
- 默认使用当前会话里的主代理完成：阅读代码、做局部搜索、实现改动、必要的解释。
- 除非触发条件满足，不派发额外子代理。

### 可选概念：验证门禁（Verification Gate）
- 它不是“常驻子代理”，而是一套流程步骤。
- 由主编排代理执行（可串行跑命令、对照清单、记录结果），确保满足 `完成条件`。

### 条件性可选：只读 Explore（Read-only Explore）
- 仅当主代理明确遇到“定位困难”时启用（例如：需要跨多目录找入口但短时间内无法确认挂载点）。
- Explore 的职责是：帮主代理找到“下一步该去哪里看/改”，而不是替主代理做实现。

## 2. 并行策略（硬规则）
1. **默认禁止并行 subagent**：同一任务同一时间不超过一个在跑的 subagent（除了极少数被文档允许的例外）。
2. **极少数例外（如未来需要）**：当两个子任务之间不存在抢占文件的风险、并且主代理已完成关键依赖定位时，才允许并行；每次例外都必须在任务说明里写清“不可冲突边界”。

> 以当前仓库节奏与维护成本考虑，建议第一阶段直接坚持“默认禁止并行”，把复杂度留到未来。

## 3. 完成门禁（C：少返工）

在任何“完成/可合并”之前，必须满足：
1. **行为门禁**：功能是否满足设计文档中的成功标准（至少一个可观测结果，比如 HUD/飘字/回合推进/阈值事件提示）。
2. **一致性门禁**：与现有系统的接缝是否正确（例如输入事件流、战斗系统状态机、波次/奖励触发时序）。
3. **回归门禁**：不破坏已有协议/战斗循环的最小闭环（例如：反应消费是否仍能被触发、波次是否仍能推进）。
4. **证据记录**：门禁执行结果要能被复盘（命令输出摘要、检查点截图/日志片段、关键断言）。

团队若采用自动化脚本（例如 `npm run ...` / `tsc --noEmit` / 构建或运行命令），可把对应命令写在门禁条目中；本设计文档先保持“由负责人补齐可执行命令”。

## 4. 何时触发 Explore（只读）
满足任一条即可触发：
1. 主代理尝试定位关键入口（Controller/系统挂载点/事件分发）后，多次仍找不到；
2. 需要理解的行为跨越多个目录且命名不一致，导致人工搜索成本显著增大；
3. 任务要求“找实现位置”而非“写新实现”，例如：某个协议/反应/状态机变量究竟在何处消费。

Explore 输出必须回到主代理，用于确定下一步要读哪些文件或改哪些模块；不直接提交实现。

## 5. 与代码审查的关系（可选）
- 若 PR/变更影响文件数量较大或涉及核心时序，建议在门禁通过前额外走一次 code review（由主代理或评审执行）。
- 小改动可跳过 reviewer，但门禁仍必须通过。

## 6. 验证与落地清单（D：文档可执行）
1. 将本文档纳入团队约定：在新任务启动时先读“最小集合 + 并行策略 + 完成门禁”。
2. 若未来团队发现 explore/shell 并行更省时，再以“例外条目”形式更新本文档，保持原则不被随意扩散。
