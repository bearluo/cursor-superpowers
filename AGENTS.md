# Minimal Subagent Workflow (A+C+D)

本文档给出你在本仓库里使用 Cursor/Agent 时的“最小 subagent 组合”约束，目标是：
- A：省调用（默认不并行派发多个子代理）
- C：少返工（完成/可合并前必须过“完成门禁”）
- D：文档可执行（规则写清楚，团队成员按清单做）

设计细节见：`docs/pipeline/2026-03-20-minimal-subagents-design.md`

## 最小 Subagent 集合

### 1) Main（主编排代理）
- 负责阅读代码、定位关键入口、实施改动、做必要的本地搜索与说明。
- 默认情况下，不额外并行派发第二个子代理去“替主做实现”。

### 2) Verification Gate（完成门禁流程）
- 不是常驻子代理，而是一套步骤：由 Main 执行并留证据。
- 任何“宣称完成/可合并”之前必须满足。
- Verification Gate 的执行/产出必须逐条覆盖下文“完成门禁（C：少返工）”的四项条目：行为门禁、一致性门禁、回归门禁、证据记录。

### 3) Explore（只读、条件性启用）
- 仅在 Main 无法快速定位关键入口/接缝点时启用。
- Explore 的输出必须回灌给 Main：下一步读哪些文件、改哪些模块；不得直接提交实现。

## 并行策略（硬规则）

1. 默认禁止并行 subagent：同一任务同一时间不超过一个在运行。
2. 例外只允许在文档明确写明“不可冲突边界”的极少数情况（可判定规则）：并行子任务仅允许“只读探索/信息收集/文档整理/验证运行”，不得同时修改并提交同一关键代码路径或同一文件；且 Main 已完成关键依赖定位（已确定将要接触的入口/状态机/文件范围）——判定标准是 Main 在例外说明中列出“本任务的写入文件/模块清单”，并证明并行子任务的写入范围为空（只有证据/信息）。若无法写出该清单，则例外判定失败，必须串行。

## 完成门禁（C：少返工）

在宣称完成/可合并之前，必须满足：
1. 行为门禁：满足设计文档的成功标准（可观测结果）。
2. 一致性门禁：接缝正确（输入事件流/状态机/触发时序）。
3. 回归门禁：不破坏最小闭环（例如协议消费仍能触发、波次仍能推进）。
4. 证据记录：可复盘证据（命令输出摘要、关键日志片段、必要截图/断言；尽量附可复现的命令/脚本片段），并放在本次任务的“Verification”部分（例如 PR 描述或提交说明）。

交付物最低要求（用于落地 D）：必须固定字段顺序（用于落款/可复制粘贴）：
- 成功观测结果（1 句）
- 关键证据（1-2 条；尽量可复现的命令/日志片段）
- 未通过原因（仅当未通过）
- 待补验证（仅当未通过；写明下一步要验证什么）

## 触发 Explore（何时启用）

满足任一条即可触发：
- Main 已多次尝试定位关键入口但仍不确定。
- 需要理解的行为跨多个目录且命名不一致，导致人工搜索成本显著增大。
- 任务本质是“找实现位置/接缝点”，而不是新增复杂实现。

Explore 的最小交付给 Main 输出模板（3-5 行）：
- 结论：找到的入口/接缝点是什么
- 证据：涉及的文件路径 + 关键日志/断言（或可复现命令输出）
- 下一步建议：Main 接下来应读取哪些文件、修改哪些模块（用清单句式）

## 代码审查（可选）

若改动影响核心时序或涉及较多文件：建议在完成门禁通过前额外走一次 code review。
